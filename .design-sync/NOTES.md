# design-sync notes

## Why this repo needed a workaround (read before re-syncing)

This repo is a personal Vite app (`private: true`), not a published component-library package — there's no `module`/`main`/`exports` in `package.json` and no library build. The converter runs in **synth-entry mode**, with three layers of workaround on top:

- **No `.d.ts`** — component discovery comes entirely from `cfg.componentSrcMap` pins (15 names), not from a shipped types tree. `.d.ts` prop extraction is therefore weaker than a real build would give.
- **CSS doesn't compile standalone.** Every component does `import './x.scss'`, but none of those `.scss` files `@use` their own variables — they rely on a Vite-only mechanism (`vite.config.ts`'s `css.preprocessorOptions.scss.additionalData`) that auto-prepends `@use "_variables.scss" as *;` to every `.scss` file at Vite build time. Raw esbuild (what the converter bundles with) has no Sass support and no equivalent auto-injection, so bundling the real source files directly would fail outright.
- **`react-router-dom` and `react-i18next` need a shared module instance.** `Header` (`NavLink`), `Home`/`Button`'s `as="link"`/`as="navlink"` forms, and every page-level component's `useTranslation()` calls all need real router/i18n context. A preview-local provider bundled separately from the main `_ds_bundle.js` would get its *own* copy of `react-router-dom` with a different React Context identity (same class of problem `reactShim` solves for `react`/`react-dom`, just not extended to other packages) — see "Router + i18n wiring" below for the fix.

**The CSS/entry fix**: `.design-sync/scripts/precompile-scss.mjs` reproduces Vite's prepend step and compiles each component's `.scss` (plus the global `reset+typography` stylesheet) to real `.css` via the `sass` package, writing output under `.design-sync/synth-src/`. `.design-sync/synth-src/<name>/index.tsx` are copies of the real source files with the stylesheet import swapped from `.scss` to the precompiled `.css`, **and any cross-component import (e.g. `Header` importing `Button`) rewritten from the `@/` alias to a relative import of the sibling synth-src copy** (`../button/index.tsx`) — importing the real `@/components/ui/button` would pull its un-rewritten `.scss` import back into the main bundle entry and break the build. Data/types/helpers/context imports (JSON data, `@/locales/types`, `@/helpers/typedEntries`, `@/context/theme/useTheme`, `@/assets/profile-photo.png`) are safe via the real `@/` alias — they carry no CSS. `.design-sync/synth-entry.mjs` re-exports all 15 from there and is the build's `cfg.entry`.

**Router + i18n wiring**: `cfg.extraEntries` merges `react-router-dom` (bare specifier) and `./src/i18n/i18n.ts` (the real i18n bootstrap module, side-effect only — its top-level `i18n.use(initReactI18next).init(...)` call runs once when the bundle loads) into the **same** `window.JeissonRodriguezResume` bundle as every component. `cfg.provider: {"component": "MemoryRouter"}` wraps every preview card in `<MemoryRouter>`, so `Header`'s `NavLink`, `Button`'s `as="link"`/`as="navlink"`, and every `useTranslation()` call all resolve against real, shared context — no cross-bundle mismatch. The generated README's auto-provider section also tells the design agent to wrap its own compositions in `<MemoryRouter>`; `.design-sync/conventions.md` explains the i18n half (which has no JSX wrapper to auto-document, since it's a side-effect import).

**To re-sync after the real components change**: re-run `node .design-sync/scripts/precompile-scss.mjs`, then re-copy the upstream logic changes into the matching `.design-sync/synth-src/<name>/index.tsx` file (the CSS-fixed copies are NOT auto-regenerated from `src/` — only their stylesheets are). A diff between the real source file and its synth-src copy should show nothing but the stylesheet import and any `@/component-alias` → `../sibling/index.tsx` rewrites.

## Scope decisions

- **`TableOfContents`/`SmartToc` is excluded.** It requires three of this app's own context providers (`react-i18next`, `FabProvider`, `ModalProvider`) to render at all — it's page-specific glue (scroll-spy + this site's own modal/FAB), not a portable design-system component. `cfg.componentSrcMap` pins it to `null`.
- **The `About` page wrapper is excluded.** It only adds the excluded `TableOfContents` around `AboutSection`/`ExperienceSection`/`EducationSection`/`SkillsSection`, which are synced individually instead.
- **Synced**: the 7 `src/components/ui/` primitives (Button, Tag, Modal, Fab, Carousel, Logo, Icon), `Header`/`Footer` (`src/components/layout/`), and `Home`/`Projects`/`AboutSection`/`ExperienceSection`/`EducationSection`/`SkillsSection` (`src/pages/`) — 15 components total, grouped as `general`/`layout`/`pages`/`about` (auto-derived from each component's real source directory).

## Known render warns (re-syncs: don't treat these as new)

- **`Icon` glyphs render as empty boxes**, and so does anything composing them (`Footer`'s social-link row renders with no visible chrome at all — ghost buttons have no background/border by design, and the icon glyph inside is invisible). `Icon` renders `<use href="/icons.svg#<id>-icon">` against this site's own `public/icons.svg` sprite. That sprite isn't part of the uploaded bundle and nothing in the claude.ai/design preview iframe serves it at that path. Labels/layout/copy around the icon are real and correct. Graded "good" on every component that hits this — see the relevant `.design-sync/.cache/review/<Name>.grade.json` for the reasoning.
- **`Modal`/`Fab` previews are visually clipped in local validation, by design.** Both are `position: fixed` overlays (`inset: 0` for Modal, viewport-corner-anchored for Fab). The converter's `cardMode: "single"` wrapper makes itself the containing block for `position: fixed` descendants via a CSS transform — but the wrapper has no intrinsic height, so its `align-items: center` / inset math collapses toward a zero-height anchor in `package-validate.mjs`'s local screenshot tool, clipping content above the fold (most visible on Modal's title). The real claude.ai/design card sizes the container from the `viewport` override (`Fab`: 320×160, `Modal`: 480×500 with `primaryStory: "Confirm"`), which a real product card render should size correctly even though the local capture doesn't. Triaged as benign; not re-chased on re-sync. `package-validate.mjs` prints `[RENDER_THIN]` for `Modal` every run — expected.
- **`Carousel`'s photo 404s** (and so does `Projects`' "api-rest" card, which renders one). `src/data/projects.json`'s `api-rest` entry references `api.png`/`api-2.png` under `/projects/`, but `public/projects/` isn't populated in this repo yet — this is a real gap in the live site's own data today, not a sync artifact. The preview uses the exact real data; nav arrows/dots/frame/card chrome all render correctly, the `<img>` itself shows the browser's broken-image icon.

## Re-sync risks

- The `.design-sync/synth-src/` copies are **hand-maintained mirrors**, not generated — if a real source file changes (new props, new markup, a new cross-component import), the synth-src copy will silently drift unless someone updates it too, including re-applying the `@/component-alias` → relative-sibling-import rewrite for any new component-to-component import. Diff the two on every re-sync.
- `.design-sync/scripts/precompile-scss.mjs` hardcodes 15 component `.scss` paths plus `src/styles/base/globals.scss` — a new component needs a line added there AND a `componentSrcMap`/synth-src/synth-entry entry, or its styles won't ship.
- The font-url rewrite in `precompile-scss.mjs` assumes fonts live under `src/assets/fonts/` and are referenced from `typography.scss` as `/src/assets/fonts/...` — if that path convention changes, the regex needs updating or `[FONT_MISSING]`/dangling `@font-face` will silently degrade brand fonts to system fallbacks.
- `cfg.extraEntries: ["react-router-dom", "./src/i18n/i18n.ts"]` bundles real `i18next` resources (all locale JSON) into `_ds_bundle.js` — any new locale namespace or key used by a future synced component needs no extra wiring (it's already in the bundle), but the bundle will keep growing with the full locale set regardless of which components are in scope (~2.8 MB already, well under the 5 MB per-file upload cap, but worth knowing if more components are added).
- No playwright-render-checked confirmation of the *actual* claude.ai/design card sizing behavior for the Modal/Fab overlay clipping noted above — only inferred from how `cardMode`/`viewport` are documented to work. Worth a visual sanity check in the live project after first open.
- The live project already has app-generated content beyond what this sync manages (`templates/`, `screenshots/`, `_ds_manifest.json`, `_adherence.oxlintrc.json`, etc.) — confirmed present via `list_files` before the second sync and deliberately left untouched (outside the `components/`/`tokens/`/`fonts/`/`_vendor/`/`_preview/`/`guidelines/` reconciliation scope). Never widen the upload `deletes` glob to cover the project root.
